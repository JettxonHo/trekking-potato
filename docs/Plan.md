## 0. 红队 v0.1 击穿的核心修正（先讲清楚改了什么）

| v0.1 缺陷 | v0.2 修正 |
|-----------|----------|
| **海拔系统性缺失**（最危险一击） | 海拔升为一等公民：P1 返回经纬度+海拔，P2 传 elevation 查天气，P4 规则按真实海拔触发 |
| **geocode API 选型错误**（山名不是地址） | 改用高德 POI 关键字搜索 + 类别筛选；内置表兜底 |
| **降级模板比报错危险**（隐藏失败） | 降级时 UI 显著标注"AI 生成失败，基础参考"，风险栏强制为空+标红 |
| **天数输入缺失**（规则依赖天数） | 输入页补"天数/返回日期"，多日徒步装备才能生成 |
| **GLM JSON 能力当开放问题** | 前置为 P0-pre 调研，决定 P5 是否启用 response_format |
| **串行调用链超时** | weather + sunEvents 用 Promise.all 并行 |
| **Astral 是 Python 库** | 改用 suncalc（纯 JS） |
| **纬度维度缺失** | gear-rules 增加纬度带（干湿/紫外） |
| **模糊匹配逻辑缺失** | 内置表用包含匹配+编辑距离兜底 |

## 1. 数据模型（海拔为一等公民）

整条链路的核心数据结构，海拔是必传字段，不是可选：

```
路线输入 → resolveLocation(route) → { name, lat, lon, elevation, source }
                                              │
            ┌─────────────────────────────────┼─────────────────────────┐
            ▼                                 ▼                         ▼
   fetchWeather(lat,lon,                     calcSunEvents              gear-rules
       elevation, date)                      (lat,lon,date)             (season, elevation,
            │                                                           days, latitudeBand)
            ▼                                                                 │
   {days:[{tempMin,tempMax,                                                   │
    precipProb,windMs}],                                                      │
    source,fetchedAt}                                                         │
            │                                                                 │
            └─────────────────────┬─────────────────────────────────────────┘
                                  ▼
                          callGLM(weather, gear, sun, route, date, level, days)
                                  ▼
                          JSON schema 校验 → 降级模板
                                  ▼
                          前端渲染（含海拔标注）
```

**海拔数据源**（两层兜底）：
1. 内置热门路线表：手填准海拔（金顶 1918m、二峰 5276m 等）
2. 未命中：查 Open-Meteo elevation API（`https://api.open-meteo.com/v1/elevation`，免费，返回该经纬度地形海拔）
3. 都失败：明确报错"未获取海拔，无法生成可靠建议"

## 3. 实现顺序（垂直切片）

### P0-pre：GLM-4-Flash JSON 能力调研（前置，非编码）
**目标**：确定 P5 的 JSON 稳定性方案，避免咽喉切片返工。
- 查智谱文档：GLM-4-Flash 是否支持 `response_format: json_object`
- 如支持：P5 启用此参数，schema 校验作为二道防线
- 如不支持：P5 纯靠 prompt 约束 + 严格 schema 校验 + 降级模板（dropout 率 5-15% 可接受，自用阶段）
- **实测（v0.3，红队击穿）**：不只查文档，用 5-10 条真实 prompt（含天气+装备规则+安全约束）实测 GLM-4-Flash 的 JSON 成功率和字段完整率。文档"支持"≠ 实际可靠，Flash 级模型指令遵循率可能低。
- 顺便确认：GLM-4-Flash 的 timeout 上限、并发限制
**验证**：结论写入 prompt.js 注释，作为 P5 实现依据。

### P0：项目骨架 + 云开发环境（地基）
**目标**：能跑起来空的小程序 + 云函数，验证开发链路通。
- 初始化 `trekking-potato/` 目录结构（按 Spec §3）
- 微信开发者工具配置 AppID + 云开发环境
- 创建空的 `getAdvice` 云函数（返回固定 mock 数据，含 elevation 字段）
- 创建空的 index 页（按钮调用云函数，弹出 mock 结果）
- git init，首个 commit，打 tag `v0.1-spec`
**验证**：真机预览，点按钮能看到 mock 返回 → 链路通。

### P1：地理编码模块（geocode.js）—— 含海拔
**目标**：路线名 → {经纬度, 海拔, name, source}。
- 实现 `resolveLocation(route)`：
  1. 先查内置热门路线表（10-15 条，含准海拔），模糊匹配（包含+编辑距离）
  2. 未命中调高德 **POI 关键字搜索**（不是 geocode）+ 类别筛选（风景名胜/自然地物）
  3. POI 命中后查 Open-Meteo elevation API 取海拔
  4. 全失败：明确报错"未找到位置"
- 编写内置热门路线表（武功山金顶1918m、四姑娘山二峰5276m、五台山朝台、喀纳斯、格聂等）
- 坐标系转换（v0.3 新增，红队击穿点）：高德 POI 返回 GCJ-02，Open-Meteo 用 WGS84。GCJ-02 偏移 100-600m，在山脊线上 500m 偏移会导致海拔查到隔壁山谷（偏差 100-300m）。**必须做 GCJ-02→WGS84 转换**后再传给 Open-Meteo。引入 `gcoord` 或手写转换算法（几十行，成熟公开算法）。这是海拔一等公民的可信基础。
**验证**：本地调试，输入"武功山"返回 {lat,lon,elevation:1918}；输入"不存在的山"报错；输入"武功山东北线"模糊匹配到主峰并提示用户确认。

### P2：天气模块（Open-Meteo，含 elevation）
**目标**：经纬度+海拔+日期 → 7 天天气，按真实海拔而非网格点。
- 实现 `fetchWeather(lat, lon, elevation, date)`：调 Open-Meteo Forecast API，**传 elevation 参数**
- 结构化：每天 tempMin/tempMax/precipProb/windMs，按传入海拔修正温度（Open-Meteo 内部处理）
- **递减率局限标注（v0.3，红队击穿）**：Open-Meteo 用标准递减率（-6.5°C/km）线性修正，在逆温层（冬季山谷比山顶更冷）、夜间辐射冷却场景 **dangerously wrong**。结果页必须标注"山区微气候无法精确预测，逆温层/辐射冷却场景温度可能反向偏差"。这不是 bug 是模型局限，诚实标注是唯一解。
- 截断 7 天，第 5 天后标注置信度递减（UI 用"参考"而非数值）
- precipProb 在中国区域标注"来自 GFS 集合，验证度低于欧美"（诚实标注）
- API 超时/失败返回明确错误
**验证**：本地调试，输入四姑娘山二峰坐标+海拔5276，返回温度应明显低于网格点（3000m）温度，验证海拔修正生效。

### P3：天文时刻模块（suncalc，纯 JS）
**目标**：经纬度+日期 → 日出日落/黄金/蓝调时刻。
- 引入 `suncalc`（纯 JS，无浏览器依赖，云函数兼容）
- 实现 `calcSunEvents(lat, lon, date)`：返回 sunrise/sunset/goldenHour/blueHour
- 海拔对日出日落的影响（5000m 约 7-15 分钟偏差，非 3-5 分钟）：MVP 不修正
- **地形遮挡（v0.3，红队击穿，比海拔更重要）**：suncalc 算的是几何日出（假设平坦地平线）。山谷中实际日出可能比天文日出晚 1-2 小时（山脊遮挡）。结果页标注"未考虑地形遮挡，山谷实际日出可能晚 1-2 小时"
**验证**：本地调试，输入武功山+7月5日，与 timeanddate.com 对比验证。

### P4：装备规则表 + Prompt 工程（含纬度带）
**目标**：LLM grounding 数据 + 系统提示词就位。
- 编写 `gear-rules.js`：季节×海拔×天数×**纬度带**规则表
  - 季节（按温度区间）：夏/过渡季/冬
  - 海拔：低<1500m/中1500-3500m/高3500-5500m/极高>5500m
  - 天数：单日/多日
  - **纬度带（新增）**：南方湿润（云贵川）/ 北方干燥（新疆内蒙青藏）——影响紫外、干燥度、温差
  - 致命风险硬约束：对应海拔/季节的致命风险装备必入清单
  - 边界处理：边界月份/海拔用区间+备注（如"5月底武功山按过渡季偏夏"）
- 编写 `prompt.js`：system prompt（角色/知识边界/JSON schema 约束/安全护栏）+ 模板
- 定义降级模板（校验失败时返回预置结构，**风险栏强制为空**）
**验证**：prompt 模板渲染变量正确注入；gear-rules 覆盖所有维度组合；纬度带区分云南4000m和新疆4000m。

### P5：核心云函数 getAdvice（咽喉切片）—— 并行 + 严格校验
**目标**：输入 → 完整路书 JSON，含 schema 校验 + 并行优化。
- 串联：resolveLocation → **Promise.all([fetchWeather, calcSunEvents])** → callGLM
- callGLM：调 GLM-4-Flash，传 system+user prompt（含 weather+gear+sun+route+date+level+days）
- 如 P0-pre 确认支持 response_format:json_object 则启用
- **JSON schema 校验**（硬约束）：
  - 所有顶层字段必须存在
  - gear 三层数组必须存在（可空）
  - risks 为数组
  - **校验严格度分层**：核心字段（weather/gear/risks）缺失→降级；非核心字段（notes/photoTiming）缺失→用默认值填充不降级
- **降级模板（重设）**：
  - 校验失败返回预置建议 + 原始天气数据
  - **降级标志位 `degraded: true`**，前端据此标红
  - **风险栏强制为空**（降级时 LLM 不可信，不输出风险建议）
  - UI 顶部显著标注"AI 生成失败，以下为基础参考，请查阅专业路书"
- 超时阈值（v0.3 修正，红队击穿）：**25 秒**。8s 在真实环境（冷启动 3-8s + resolveLocation 4-6s + GLM 3-8s）暖态就超时、冷态必崩。25s 给真实链路留余量。
- **前端分步加载（v0.3 新增）**：云函数先返回地理+天气+天文（快，<8s），前端立即渲染；GLM 结果异步返回后增量更新建议部分。让用户在等待中有反馈。
- 注入 meta（generatedAt、weatherSource、llmModel、**elevation**）
**验证**：端到端跑通"武功山/7月5日/单日/中级"；构造 LLM 异常返回，验证降级模板 + degraded 标志；构造超时，验证 25s 报错；验证分步加载：天气先出、建议后出。

### P6：前端输入页（index）—— 补天数
**目标**：用户输入路线/出发日期/**天数**/水平并提交。
- 输入表单：路线名（文本）、出发日期（日期选择器）、**天数**（v0.3 改：数字输入框 1-7，而非粗粒度分类——2天和3天装备差异是实质性的）、徒步水平（三级单选）
- 输入校验：空路线、过去日期、未选水平/天数兜底提示
- 提交按钮：调用 getAdvice，loading 态
- 错误态：云函数失败友好提示（非白屏）
**验证**：真机预览，填表（含天数）提交，看到 loading→跳转结果页。

### P7：前端结果页（result）—— 含海拔标注 + 降级标红
**目标**：渲染路书 JSON，海拔透明，降级显著。
- 顶部标注"本数据基于海拔 XXXXm"（海拔透明化）
- 天气窗口卡片（7 天，第 5 天后标"参考"）
- 装备清单（必备/推荐/可选分层）
- 风险提示（致命风险红色加粗；**降级时此栏标红"AI 不可用，请查专业路书"**）
- 出片时机（标注"海平面参考值"）
- 微气候原始数据（湿度/风速/露点差）
- 免责声明（底部固定）+ 数据来源时间戳 + 海拔
- **降级态**：degraded=true 时，顶部红色横幅"AI 生成失败，基础参考"
**验证**：真机预览，正常 JSON 渲染全字段；传 degraded=true，验证标红横幅；传残缺 JSON，验证不崩溃。

### P8：集成测试 + 部署（含专业 review）
**目标**：端到端跑通 3 条真实路线，部署上线。
- 3 条路线：武功山（低海拔雷暴）、四姑娘山二峰（高海拔技术）、五台山朝台（多日）
- **非 happy path 测试（v0.3 新增，红队击穿）**：加 2 条非内置表路线（如"小五台""大白山"），验证高德 POI + Open-Meteo elevation 兜底链路；加 1 条 Open-Meteo 慢响应场景（mock 延迟 8s），验证分步加载+超时
- **与公开路书对比**：不只看"没报错"，重点查**遗漏型错误**（如二峰没提冰爪）
- **专业 review**：至少 1 条路线找有经验的驴友/领队 review，查 LLM 不知道的风险
- 异常输入测试：空路线、过去日期、无效水平、小众地名、边界海拔
- 海拔验证：四姑娘山二峰温度应明显低于网格点，证明海拔修正生效
- 云函数部署云端，真机全链路验收，打 tag `v0.3-mvp`
**验证**：3 条路线真机跑通且建议合理（含专业 review）；海拔修正可见；异常输入有兜底；朋友可扫码使用。

## 4. 并行 / 串行划分（修正）

**严格串行**：P0-pre（调研）→ P0（骨架）→ P1（地理）→ P5（核心）→ P6→P7→P8

**P1 完成后可并行**：
- P2（天气，依赖 P1 的 coords+elevation）
- P3（天文，依赖 P1 的 coords）
- P4（装备规则，独立于 P1-P3）

注：v0.1 写的"P2/P3 独立可并行"不准确——两者都依赖 P1 输出。v0.2 修正为"P1 完成后 P2/P3/P4 可并行"。

## 5. 风险与缓解（修正）

| 风险 | 缓解 |
|------|------|
| GLM-4-Flash JSON 不稳 | P0-pre 前置调研；P5 严格 schema 校验 + 降级（标红，非隐藏） |
| 高德 POI 对山名匹配差 | P1 内置表兜底 + 模糊匹配 + 用户确认 |
| **海拔数据缺失（最危险）** | P1 内置表准海拔 + Open-Meteo elevation API；全链路 elevation 必传 |
| Open-Meteo 网格点海拔≠真实海拔 | P2 传 elevation 参数请求该海拔天气 |
| 山区微气候不可建模 | 结果页诚实标注"山区微气候无法精确预测，仅供参考" |
| 云函数冷启动 + 串行超时 | weather/sunEvents 用 Promise.all；超时 25s+分步加载（天气先出）；**Open-Meteo 从腾讯云连通性 P2 需实测（国际 API，延迟/稳定性未验证）** |
| 降级隐藏失败 | degraded 标志 + UI 标红 + 风险栏强制空 |
| 单人开发流程自律 | P0 即 git init，每切片完成才合并 dev |

## 6. 开放问题（v0.2 剩余）

1. ~~Astral 兼容性~~ → 已定：改用 suncalc（纯 JS）
2. ~~GLM response_format~~ → 移至 P0-pre 前置调研

  1. 先查内置热门路线表（10-15 条，含准海拔），模糊匹配（包含匹配优先；编辑距离≤2 时强制用户确认，防止"武当山/武功山""黄山/庐山"假阳性）
