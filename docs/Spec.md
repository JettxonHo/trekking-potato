# 第一部分：PRD（产品需求）

## 1. 产品定位

**徒步薯** —— 一个垂直于徒步领域的 AI 工具。
用户输入行程参数，Agent 输出结构化建议（路书格式）。
它是工具，不是内容平台；是 agent，不是攻略库。

核心边界：
- 做行前决策辅助，不做行中导航
- 做建议生成，不做内容分发
- 做垂直工具，不做综合平台

## 2. 目标用户

| 阶段 | 用户 | 场景 |
|------|------|------|
| MVP / 自用 | 自己 + 朋友圈驴友（5-10 人） | 计划徒步时打开小程序查建议 |
| 公开上线 | 国内跨省驴友 | 出行前查路线建议、天气窗口、装备 |

## 3. 核心场景（User Story）

> 作为一条典型路径：用户计划周末去武功山 → 打开小程序 → 输入"武功山 / 7月5日 / 中级" → Agent 返回天气窗口 + 装备清单 + 风险提示 + 注意事项 → 用户参考做行前决策。

输入参数：
- 路线名或地点（如"武功山"/"四姑娘山二峰"）
- 出发日期（如 2026-07-05）
- 徒步水平（初级 / 中级 / 高级）

输出结构（路书格式）：
- **天气窗口**：未来 7 天逐日温度区间、降水概率、风速、推荐出发时段
- **装备清单**：按必备 / 推荐 / 可选分层，每项配一句理由
- **风险提示**：该路线该季节核心风险（如雷暴、失温、高反）
- **注意事项**：路况、补给、许可、紧急联系
- **出片时机**：日出/日落时间、黄金时刻、蓝调时刻、云海概率提示（基于湿度/风速估算，标注为参考）
- **微气候原始数据**：湿度、风速、温度露点差（供用户自行判断云海可能性；不输出"云海概率"结论——云海形成依赖逆温层等地形抬升因素，仅凭地表数据无法可靠预测）
- **免责声明 + 数据来源时间戳**

## 4. MVP 功能范围

**做：**
- 单页输入表单（路线 / 日期 / 水平）
- 调用 LLM + 天气 API 生成结构化建议
- 建议结果页展示（路书格式）
- 免责声明 + 数据来源标注

**不做（明确排除）：**
- 账号体系 / 登录注册（自用阶段不必要）
- 历史记录持久化（后续版本）
- 路线推荐 / "去哪"建议（Stage 1+）
- 社交 / UGC / 评论（不做平台）
- 地图导航 / 轨迹记录（不做行中）
- 实时天气推送（MVP 只做行前一次性查询）
- 付费 / 订阅（自用免费）

## 5. 成功标准（Stage 0 验证）

核心假设：**徒步薯对自用用户有实际决策价值。**

验证方式：MVP 上线后，发给 5-10 个朋友试用，采集信号（自用阶段不引入追踪持久化，口头反馈即信号；公开上线后再切 openId 匿名追踪，写进后续 Spec 升级）：

| 指标 | 通过门槛 |
|------|---------|
| 主观有用性 | >70% 试用者回答"建议有用"（小样本信号，非统计结论） |
| 复用意愿 | 作为信号采集，不作硬门槛（无追踪持久化无法区分真复用与客套；自用阶段这点精度不值得多加持久化层） |
| 缺口反馈 | 收集到的"缺什么"清单，作为 v2 输入 |

补充技术指标：
- 建议生成时间 < 20 秒（含地理编码 + 天气 API + LLM 调用；超时降级为预置模板+原始天气数据，不让用户等白屏）—— 目标值，待真实 benchmark
- 天气数据与 Open-Meteo 一致（数据源透明）
- 小程序冷启动 < 3 秒

## 6. 非功能需求

- **成本**：自用阶段月成本 < 10 元（GLM-4-Flash 免费 + 云开发免费额度 + Open-Meteo 免费）
- **可用性**：非商业承诺，偶发不可用可接受
- **安全责任**：明确免责，建议仅供参考，不替代官方气象/轨迹信息

# 第二部分：技术 Spec

## 1. 技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | Taro 4 + React 18（编译为微信小程序） | 原生 WXSS 引擎反复 LifeCycle.load 崩溃，Taro 编译产物零 undefined；用户决策方案 A |
| 后端 | 微信云开发（云函数 Node.js） | 零运维、原生集成微信登录、自用免费额度 |
| 数据库 | 云开发云数据库（NoSQL） | MVP 不强需，预留用于未来历史记录 |
| LLM | 智谱 GLM-4.7（含 response_format） | Flash 0/10 schema 通过率不可用，4.5 限流，4.7 带 response_format 3/3 通过；用户有 coding plan |
| 天气 | Open-Meteo Forecast API | 免费、无需 key、任意经纬度 7-16 天预报 |
| 地理编码 | 高德地理编码 API（主）+ 内置热门路线表（兜底） | 高德个人免费 5000次/日、精确 POI、原生 GCJ-02 合规；内置表应对接口失败 |
| 天文时刻 | sun-event / Astral 库（离线计算） | 日出日落、黄金/蓝调时刻，确定性天文公式，零 API 成本 |
| 凭据管理 | 云函数环境变量 | API key 不进前端、不进代码仓库 |

**地图合规说明**：中国地图坐标必须经 GCJ-02 加密（法律强制）。高德原生输出 GCJ-02；Open-Meteo 用 WGS-84，用于天气查询（非地图渲染）无偏移问题。MVP 不渲染轨迹地图，仅做经纬度定位拉天气，合规风险低。

## 4. 代码风格

云函数核心调用示例（体现风格）：

```javascript
// cloudfunctions/getAdvice/index.js
const cloud = require('wx-server-sdk')
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

exports.main = async (event) => {
  // 1. 校验输入
  const { route, date, level } = event
  if (!route || !date || !level) {
    return { ok: false, error: 'missing_params' }
  }

  // 2. 路线名 → 经纬度（LLM 抽取，MVP 兜底）
  const coords = await resolveCoords(route)

  // 3. 拉天气
  const weather = await fetchWeather(coords, date)

  // 4. 调 LLM 生成建议
  const advice = await callGLM({ route, date, level, weather })

  // 5. 注入数据来源与时间戳
  return {
    ok: true,
    data: {
      ...advice,
      weather,
      meta: {
        generatedAt: new Date().toISOString(),
        weatherSource: 'Open-Meteo',
        llmModel: 'glm-4-flash',
      },
    },
  }
}
```

命名约定：
- 文件：`camelCase`（JS）/ `kebab-case`（页面目录）
- 云函数：动词开头（`getAdvice`、`resolveCoords`）
- 常量：`UPPER_SNAKE_CASE`
- 注释：只在非自明处写，解释"为什么"不写"是什么"

## 7. 边界（Always / Ask First / Never）

**Always：**
- 调 LLM 前校验输入完整性
- 天气数据标注来源（Open-Meteo）和时间戳
- 结果页底部固定显示免责声明
- API key 只放云函数环境变量

**Ask First：**
- 更换 LLM 模型（Flash → GLM-4）
- 新增数据源（如接入和风天气）
- 改云数据库 schema
- 引入新依赖

**Never：**
- 把任何 API key 放前端代码或仓库
- 上线未本地调试过的云函数
- 删除免责声明
- 对致命风险用中立措辞（必须用强建议+理由）
- 把 LLM 输出当作可信数据源直接渲染——必须有 JSON schema 校验和降级模板（解析失败时返回预置模板 + 原始天气数据，绝不白屏）
- 输出"云海概率"这类无法可靠预测的结论性字段（只展示原始微气候数据，判断留给用户）

## 8. 安全 / 信任工程

- **免责声明**（结果页固定）：本建议由 AI 生成，仅供参考；天气数据来自 Open-Meteo；出行前请核实官方气象与路线信息；户外活动有风险，责任自负。
- **数据来源标注**：每条天气数据标注 Open-Meteo + 拉取时间戳；LLM 模型版本标注。
- **致命风险措辞**：对失温、雷暴、高反等致命级风险，用强建议措辞（"必须携带防雨外套，因午后雷暴高发"），而非中立清单。
- **LLM 输出可靠性（最危险一击的修复）**：`callGLM` 之后必须接 JSON schema 校验 + 字段完整性检查；校验失败时降级为预置模板 + 原始天气数据返回，前端绝不渲染未校验的 LLM 原文。LLM 是整条链路最不可控组件，不可当可信管道。
- **API key 隔离**：智谱 API key 只存云函数环境变量，前端零感知。

## 10. LLM / Agent 架构

### 10.1 当前架构定位（名实相符）
MVP 诚实定位为 **"LLM 增强的建议工具"**，不是自主 agent。流程是确定性流水线：用户输入 → 查地理编码 → 查天气 → LLM 生成结构化建议 → schema 校验 → 返回。LLM 的角色是"结构化数据翻译成自然语言建议的生成器"，不是"自主决策行动路径的 agent"。这不丢人——一个 prompt 写得好的流水线，实际效果很可能优于设计粗糙的 ReAct agent（后者每多一轮 LLM 调用就多一次幻觉、多一次延迟、多一次成本）。

等达到真 agent 标准（自主推理场景）后再改名为 "Agent"，避免名实不符的预期偏差。

### 10.2 组件取舍（MVP 做什么、不做什么）

| 组件 | MVP | 理由 |
|------|-----|------|
| Prompt 工程 | **必须做** | 任何 LLM 应用的核心，是输出质量的根基 |
| RAG | 轻量 grounding（装备规则表注入 prompt） | 结构化 context 够用，向量检索是 v2 的事 |
| Function Calling / Tool Use | 不做 | 决策路径确定性（总要查地理编码+天气+生成），无自主决策分叉 |
| ReAct（推理-行动循环） | 不做 | 单轮生成，无多步推理需求；多轮 LLM 调用在手机端延迟不可接受 |
| MCP（Model Context Protocol） | 不做 | 仅 2 个固定工具，协议层是过度工程 |
| Harness / 编排框架 | 写死 async 流水线 | 单链路无分支，原生 async/await 够用，不引入 LangChain |

### 10.3 Prompt 工程（必须定义）

**系统提示词设计原则：**
- 角色定位：徒步行前建议助手，只做行前决策辅助，不做行中导航
- 知识边界：明确告知 LLM "你的路线知识可能过时/不全，不确定时必须声明"
- 输出格式约束：必须返回符合 §10.4 定义的 JSON schema 的 JSON，禁止返回自然语言正文
- 安全护栏：致命风险（失温/雷暴/高反/落石）必出且用强建议措辞；不确定时声明"此信息未经核实，请查阅官方/两步路"；禁止编造不存在的路线细节（里程、补给点、危险点）

**Prompt 模板结构：**
```
[System Prompt: 角色 + 知识边界 + JSON schema 约束 + 安全护栏]

[User Context]
路线：{route}
出发日期：{date}
徒步水平：{level}

[Grounding Data]
{weather JSON}      ← Open-Meteo 原始数据，含来源标注
{gear rules JSON}   ← 季节×海拔×天数规则表

请基于以上数据，生成符合 schema 的路书建议 JSON。
```

### 10.4 输出 JSON Schema（约束 LLM 格式）
```json
{
  "weatherWindow": {
    "days": [{"date":"2026-07-05","tempMin":18,"tempMax":26,"precipProb":0.6,"windMs":4,"recommendedWindow":"清晨6-9点"}],
    "source": "Open-Meteo",
    "fetchedAt": "2026-06-28T10:00:00Z"
  },
  "gear": {
    "essential": [{"item":"防雨外套","reason":"午后雷暴高发"}],
    "recommended": [{"item":"登山杖","reason":"长下坡保护膝盖"}],
```

### 10.5 Agent 演进路径（后续 Spec 升级时启用）

| 阶段 | 架构 | 触发条件 |
|------|------|---------|
| MVP（当前） | 确定性流水线 + LLM 生成器 | —— |
| Stage 1 | + Function Calling | 引入路线推荐（LLM 自主决定是否调路线库检索） |
| Stage 2 | + ReAct 多轮推理 | 复杂任务（"帮我规划3天川西路线"需要搜索→对比→选优→排日程） |
| Stage 3 | + MCP 协议层 | 工具数 >5 且动态变化（接入实时路况+装备电商+轨迹平台+气象预警） |

**何时改名 Agent**：达到 Stage 1/2（有自主决策或多步推理）时。

# 附录：与原设计文档的差异

本 Spec 基于原《徒步 AI Agent 设计方案》但做了重大精简，差异如下：

| 原设计文档 | 本 Spec | 原因 |
|-----------|---------|------|
| 公众号→小程序→App 三级演进 | 直接做小程序 | 产品是工具不是内容，不需要内容冷启动阶段 |
| Stage 0 = 公众号图文传播验证 | Stage 0 = 最小可交互小程序试用 | 工具型产品用内容指标验证是因果错配 |
| 传播策略（渠道/社交货币/种子用户） | 砍掉 | 自用阶段不谈传播 |
| 增长与商业模式章节 | 砍掉 | 自用工具不涉及 |
| 路线推荐功能（Stage 1+） | 暂不列入 Spec | 等 MVP 验证通过再讨论 |
| 装备建议强度二元规则 | 保留（致命风险用强建议） | 信任工程核心 |
| 海拔修正、可信度三层标注 | 简化 | MVP 先用基础版，后续迭代 |
